# 06 ¬∑ Testing & Acceptance Gates

The loop advances **only** on green gates (`AGENTS.md` ¬ß0). This doc is the canonical catalogue of
gate commands, thresholds, and the golden-test protocol. Tasks reference these by name.

## 1. The gate commands (npm scripts)

These scripts are established by early Phase-1 tasks. Until a script exists, the task that creates
it is a dependency ‚Äî never invent a gate.

| Script | What it runs | Pass condition |
|---|---|---|
| `npm run typecheck` | `tsc --noEmit` (strict) | 0 errors |
| `npm run lint` | ESLint (incl. import-boundary rules) | 0 errors |
| `npm test` | Vitest, all unit + golden | all pass |
| `npm test -- <path>` | scoped unit/golden | all pass |
| `npm run test:golden` | engine golden-image suite | all within tolerance (¬ß3) |
| `npm run size` | bundle-size check | within budgets (¬ß2) |
| `npm run test:e2e` | Playwright, fixtures mode | all specs pass |
| `npm run test:a11y` | axe on the modal (in e2e) | 0 serious/critical violations |
| `npm run check:no-secret` | build-time scan of browser bundle | no `sk_`/secret-key code path present |
| `npm run build` | produce browser + `/server` bundles | exits 0, outputs present |

`npm run verify` runs the standard set (`typecheck lint test size`) and is the default per-task gate
unless a task names a narrower or additional one.

## 3. Golden tests ‚Äî the definition of a correct engine port

**What they lock:** that the ported `cutout-engine` reproduces the **store's** deterministic
rendering math (`docs/05`), which is the authoritative reference (not Appendix D, not the absent
Shopify prototype). The baselines are **not** self-generated by our own engine ‚Äî that would only test
for regressions. They come from **two independent trusted sources** (GP-13):

- **(a) Hand-computed numeric anchors** ‚Äî the `docs/05` formulas are exact, so `baseWidth/baseHeight`,
  `maxScale`, `x/y`, `textOffsetY`, `lowRes` are computed by hand for each case and asserted exactly.
  A wrong port cannot "match a wrong golden" because these values are derived from the spec, not the
  engine.
- **(b) Store-rendered composite baselines** ‚Äî the store's actual render code
  (`../treatink/.../functions/canvasRenderer.js` + `renderPetName.js`) is plain JS and runnable
  headless. A one-off `test/golden/gen-baseline.mjs` executes **that** code (jsdom + `@napi-rs/canvas`,
  Mitr bundled per `docs/design-reference` ¬ß4) over the fixture matrix to emit trusted composite PNGs.
  These are the golden images. This makes the port validated against the reference implementation, not
  against itself.

Protocol:
1. **Fixture matrix.** Committed `(photo, cutout PNG, transform)` cases: portrait/landscape/square
   photos; each of the four `pet_name_position` values; a low-res photo (warning); `scale` at `1.0`,
   mid, `maxScale`; off-center pan; rotation 0 (MVP). Cutouts/photos from `../treatink/.../public/
   frames` (GP-10).
2. **Baselines.** Generate composites via (b) and numbers via (a); commit both under
   `test/golden/expected/`. The generator is committed but run manually (baselines are frozen).
3. **Port render.** The SDK engine renders each case in Node via an injected `@napi-rs/canvas`
   (`docs/09`); output = PNG + derived numbers.
4. **Compare.** PNG diff ‚â§ **0.1 %** of pixels differing by > 2 (of 255) on any channel. Numeric
   anchors: **exact** for integer pixels, `‚â§ 1e-6` for normalized floats. (If font rasterization
   across platforms pushes text tiles over tolerance, exclude the text region from the pixel diff and
   rely on the numeric text-anchor + a separate text-placement assertion ‚Äî document if applied.)
5. **Freeze.** A golden change requires a reviewed commit explaining the math change ‚Äî never a silent
   re-baseline (`AGENTS.md` ¬ß0.4).

A golden change requires an explicit, reviewed commit explaining the math change ‚Äî never silently
re-baseline to make a test pass (`AGENTS.md` ¬ß0.4).

## 4. E2E happy path (Charter ¬ß14, fixtures mode)

Playwright suite must cover, end to end, in fixtures mode:
`catalog ‚Üí designer.open ‚Üí upload (incl. one HEIC) ‚Üí position/zoom ‚Üí add personalization text ‚Üí
save (simulated upload success AND failure‚Üíretry) ‚Üí orders.buildPayload ‚Üí server submitOrder` ‚Äî
plus **draft re-open** (`designer.open({ draftId })`).

Failure injection uses `tk.fixtures.failNext('assets.finalize', { status: 422, code:
'upload_validation_failed' })` and the browser-PUT failure path (`upload_failed`).

## 5. Accessibility (Charter ¬ß7.3)

- Automated: axe in the e2e run ‚Äî 0 serious/critical on the open modal.
- Manual (human checkpoint, not a loop gate): keyboard-only pass ‚Äî focus trap, focus restoration,
  Escape to close, all controls tabbable+labeled (zoom buttons included), low-res warning announced.

## 6. Security gates (Charter ¬ß10 ‚Äî hard)

- `npm run check:no-secret`: static scan proving the **browser** bundle has no secret-key path (no
  `sk_`/`tk_live_` bearer attachment, no import of `/server`). Fails the build if found.
- Key-guard unit test: `Treatink.init` throws `key_scope_violation` for `sk_*`/`tk_live_*` and any
  non-`pk_*` key; accepts `pk_test_`/`pk_live_`.
- Persistence test: assert no image bytes are written to `localStorage`/IndexedDB (drafts hold only
  references) ‚Äî a spy/mock asserts stored values are small JSON, never Blobs/DataURLs.
- No-third-party-request test: in e2e, assert the only network origins hit are the configured
  API/CDN (and `blob:`), matching the recommended CSP (Charter ¬ß10.3).

## 7. Test layout

```
test/
  unit/            # *.test.ts alongside source is preferred; shared helpers here
  golden/
    fixtures/      # (photo, cutout, transform) inputs
    expected/      # frozen golden PNGs + numbers
    golden.test.ts
  e2e/             # Playwright specs (fixtures mode)
  a11y/            # axe integration (may live inside e2e)
fixtures/          # SHIPPED backend fixtures (catalog, assets, errors) ‚Äî Charter ¬ß11, docs/08
```

Shipped fixtures (`fixtures/`) are product data (Charter ¬ß11, Appendix B): 14 products, ~95 cutouts,
including edge cases (products without zones, slug collisions, the storefront-only Riley's product,
at least one `vertical: "merch"`). Test-only inputs live under `test/`.

## 8. CI

CI runs `npm run verify && npm run test:golden && npm run test:e2e && npm run check:no-secret` on
every push. Green CI is the phase **exit** condition (plus the per-phase extras in
`phases/00-overview.md`). CI config is a Phase-1 task.
